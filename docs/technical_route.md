# 技术路线

本文档用于记录本项目从 Linux 应用层开发到系统底层知识补强的整体技术路线。

## 项目定位

本项目定位为嵌入式 Linux 应用开发项目，优先实现可演示、可部署、可写入简历的人脸识别终端。项目主线放在应用层能力，包括摄像头采集、屏幕显示、人脸识别、多线程架构、日志记录和 systemd 服务化。

底层驱动和系统移植作为后续增强方向，用于补充对 MIPI 摄像头、MIPI 屏幕、设备树、内核编译和 media pipeline 的理解。

## 阶段一：硬件链路验证

目标：确认板端摄像头和屏幕在 Debian 12 下可用。

摄像头验证：

- 使用 `dmesg` 查看 IMX415 驱动加载日志。
- 使用 `v4l2-ctl --list-devices` 查看视频设备。
- 使用 `v4l2-ctl --list-formats-ext` 查看分辨率、格式和帧率。
- 使用 `media-ctl` 查看 media pipeline。
- 使用测试程序抓取一帧图片。

屏幕验证：

- 确认显示链路是 framebuffer、DRM/KMS 还是桌面环境。
- 显示纯色或测试图片。
- 记录屏幕分辨率、刷新率和方向。

阶段产出：

- `docs/camera_bringup.md`
- `docs/display_bringup.md`
- 测试图片或截图
- 设备节点、格式和日志记录

## 阶段二：实时预览

目标：打通摄像头采集到屏幕显示的基础图像通路。

推荐实现路线：

1. 先用 OpenCV 快速验证摄像头读取和屏幕显示。
2. 再根据效果决定是否补充 V4L2 原生采集实现。
3. 记录采集分辨率、像素格式、FPS 和显示延迟。

关键技术点：

- 摄像头设备打开
- 分辨率和像素格式设置
- 图像帧读取
- 图像格式转换
- 屏幕预览显示
- FPS 统计

## 阶段三：人脸检测和识别

目标：实现人脸检测、人脸录入、人脸识别和陌生人判断。

推荐实现路线：

1. 先接入轻量级人脸检测模型，实现画框显示。
2. 再实现人员录入，保存姓名、图片和特征。
3. 最后实现实时比对，显示姓名、置信度和识别状态。

可选推理方案：

- OpenCV DNN：适合快速验证，集成成本低。
- ONNX Runtime：模型选择更灵活，适合后续工程化。
- NCNN：轻量，适合嵌入式平台。
- 平台 NPU/RKNN 类方案：后续性能优化阶段再考虑。

## 阶段四：工程化重构

目标：把 Demo 整理成可维护的 Linux 应用项目。

建议模块划分：

- `camera`：摄像头采集
- `display`：屏幕显示
- `face`：检测、识别、人员库
- `service`：配置、日志、识别记录
- `common`：队列、时间、公共工具

建议线程模型：

- 采集线程：负责摄像头帧读取。
- 推理线程：负责人脸检测和识别。
- 显示线程：负责画面绘制和屏幕输出。
- 记录线程：负责日志和识别记录写入。

关键设计原则：

- 推理慢于采集时允许丢弃旧帧，优先保证实时性。
- 日志和数据库写入不要阻塞主识别流程。
- 配置参数不要硬编码，统一放到配置文件。

## 阶段五：服务化部署

目标：让程序具备嵌入式终端的运行形态。

计划实现：

- 编写启动脚本。
- 编写 systemd service 文件。
- 支持开机自启动。
- 支持异常退出自动重启。
- 支持 journalctl 查看运行日志。
- 支持配置文件控制设备路径、分辨率、阈值和数据目录。

## 阶段六：性能记录和优化

目标：形成可以写进 README 和简历的量化数据。

建议记录：

- 摄像头采集 FPS
- 屏幕显示 FPS
- 单帧检测耗时
- 单次识别耗时
- CPU 占用
- 内存占用
- 连续运行时间

可优化方向：

- 降低输入分辨率
- 推理线程跳帧
- 使用轻量模型
- 复用图像缓冲区
- 控制日志写入频率
- 后续尝试硬件加速推理

## 阶段七：底层知识补强

目标：补足面试中可能追问的 Linux 底层链路。

建议整理内容：

- `/dev/video*` 设备节点来源
- V4L2 应用层采集流程
- MIPI CSI、sensor、ISP、video node 的关系
- IMX415 设备树节点关键字段
- media pipeline 调试方法
- MIPI 屏幕显示链路
- 内核编译、设备树编译和镜像烧录流程

这部分不要求一开始就修改驱动，但需要能讲清楚应用层使用的设备节点背后是怎样被内核识别和暴露出来的。
